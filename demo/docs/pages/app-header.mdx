# AppHeader

Header bar with compound sub-components for search, app switching, notifications, and user profile. Drop `AppHeader` into your `AppLayout` and nest `AppHeaderSearch`, `AppSwitcher`, and `AppHeaderNotifications` inside `AppHeaderActions` for the right side. The profile trigger is [`ProfilePopover`](/profile-popover) itself — pass it `user`/`variant` for the compact header-style trigger. There's no separate `AppHeaderProfile` component.

<ExampleTabs
  title="App header with all components"
  preview={<AppHeaderPreview />}

/>

Click the profile avatar to open the profile popover with variant toggle (cycles `full` → `avatar`). The search input shows groups when you type. Notifications open on the bell click.

## Variants

`AppHeader` accepts a `variant` prop that controls its surface style.

- **`"default"`** (default) — subtle rounded panel surface, matching the rest of the `AppLayout` content area.
- **`"plain"`** — square edges (no border radius), `--gsl-primary` background, and `--gsl-on-primary` text. Used for dashboard shells where the header should read as a solid top bar rather than a panel.

## Components

### `AppHeader`

Root flex container that renders children. `AppHeaderSearch` goes on the left; everything else is typically grouped inside `AppHeaderActions` for the right side.

| Prop | Type | Default | Description |
|------|------|---------|-------------|
| `variant` | `"default" \| "plain"` | `"default"` | Visual style — see [Variants](#variants) |
| `className` | `string` | — | Merged onto the root element |
| `children` | `ReactNode` | — | Header content — search, actions, etc. |

### `AppHeaderActions`

Groups items on the right side of the header with a gap. Typically contains `AppSwitcher`, `AppHeaderNotifications`, and `ProfilePopover`.

| Prop | Type | Description |
|------|------|-------------|
| `className` | `string` | Merged onto the actions wrapper |
| `children` | `ReactNode` | Right-side items (switcher, notifications, profile) |

### `AppHeaderBranding`

Logo + title/subtitle slot for the left side of the header, typically used in place of `AppHeaderSearch` for dashboard shells (see the `"plain"` variant example above).

| Prop | Type | Description |
|------|------|-------------|
| `logo` | `ReactNode` | Inline logo node (e.g. an `<img>` or icon) |
| `title` | `ReactNode` | Main title text, rendered bold |
| `subtitle` | `ReactNode` | Secondary text, rendered larger but not bold |
| `children` | `ReactNode` | Custom content — overrides `title`/`subtitle` rendering |
| `className` | `string` | Merged onto the root element |

```tsx
<AppHeaderBranding
  logo={<img src="/logo.png" alt="" width={28} height={28} />}
  title="GSL PORTAL"
  subtitle="GSL Component Library"
/>
```

`title` is rendered in the heading font (`--gsl-branding-title-font`, `--gsl-branding-title-size`) — everything else in the branding block, including `subtitle`, stays on the body font.

For anything beyond a single title/subtitle pair — a badge next to the title, a stacked two-line layout, a custom element entirely — skip `title`/`subtitle` and pass declarative `children` instead. `AppHeaderBranding` only builds the title/subtitle markup when `children` is omitted, so `children` fully replaces it (the `logo` slot still renders independently):

<ExampleTabs
  title="Declarative branding — children override title/subtitle"

Reuse `.gsl-app-header__branding-text` / `.gsl-app-header__branding-title` on your own markup (as above) to inherit the same font/size/color tokens, or write fully custom styles.

### `AppHeaderSearch`

Data-driven search input that renders results in a `Command` popover list. Pass `data` as groups of items; the component maps them to `CommandGroup` → `CommandItem` elements.

Type to search — the component fires `onSearch` with a debounced value (300ms). Drive your data fetch from this callback and pass the result back as `data`.

| Prop | Type | Default | Description |
|------|------|---------|-------------|
| `data` | `AppHeaderSearchDataGroup[]` | — | Groups of items rendered as `CommandGroup` elements |
| `onSearch` | `(value: string) => void` | — | Debounced search callback — wire to tanstack query here |
| `showEmpty` | `boolean` | `false` | Show "no results" when search has value but no items |
| `emptyLabel` | `string` | `"No results"` | Text shown in empty state |
| `placeholder` | `string` | `"Search..."` | Input placeholder |
| `label` | `string` | `"Search"` | Accessible label for the input |
| `children` | `ReactNode` | — | Rendered inside `CommandList` after groups |
| `className` | `string` | — | Merged onto the root element |

The input has a pill shape (`--gsl-radius-xl`) and caps at `max-width: 400px`.

Use `children` for custom rendering inside the results list — useful when you need full control over the `CommandGroup` elements:

```tsx
<AppHeaderSearch onSearch={handleSearch}>
  <CommandGroup heading="Recent">
    <CommandItem onSelect={() => navigate("/reports/latest")}>
      Latest report
    </CommandItem>
  </CommandGroup>
</AppHeaderSearch>
```

### `AppHeaderNotifications`

Bell button that opens a Radix popover. Pass notification items as `children`. Use `loading` to show a skeleton placeholder while fetching.

| Prop | Type | Default | Description |
|------|------|---------|-------------|
| `loading` | `boolean` | `false` | Show 3-row skeleton pulse animation |
| `loadingLabel` | `string` | `"Loading notifications..."` | Accessible label for loading state |
| `children` | `ReactNode` | — | Rendered inside the popover body |
| `className` | `string` | — | Merged onto the trigger button |

Render each notification as an [`AppHeaderNotificationItem`](#appheadernotificationitem) — don't hand-write `.gsl-notif-popover__*` divs:

```tsx
<AppHeaderNotifications loading={notifLoading}>
  {notifications.map((n) => (
    <AppHeaderNotificationItem
      key={n.id}
      text={n.text}
      time={n.time}
      unread={n.unread}
      onClick={() => markAsRead(n.id)}
    />
  ))}
</AppHeaderNotifications>
```

### `AppHeaderNotificationItem`

A single notification row — an unread dot (or a subtle read background), message text, and a time string. Becomes keyboard-focusable (`role="button"`, Enter/Space activate it) only when `onClick` is passed; without it, it's a static, non-interactive row.

| Prop | Type | Default | Description |
|------|------|---------|-------------|
| `text` | `ReactNode` | — | **Required.** Notification message content |
| `time` | `ReactNode` | — | Relative/formatted time string (e.g. `"2m ago"`) |
| `unread` | `boolean` | `false` | Shows the leading unread dot; read items get a subtle background instead |
| `onClick` | `() => void` | — | Called when the row is clicked or activated via Enter/Space |
| `classNames` | `AppHeaderNotificationItemClassNames` | — | Part-level class overrides |
| `className` | `string` | — | Merged onto the root element |

Also accepts remaining `HTMLAttributes<HTMLDivElement>` (except `onClick`, which uses the simpler `() => void` signature above).

### Profile trigger

The profile trigger in the example above is [`ProfilePopover`](/profile-popover) with `user`/`variant` props — see that page for the full prop table (`user`, `variant`, `items`, `loading`/`loadingLabel`, `onSignOut`, `children`), the built-in theme toggle, and the loading state. `AppHeader` doesn't have its own profile component — `ProfilePopover` is the same one `SidebarFooter` uses.

## Notes

- `AppHeader` uses compound children: nest sub-components directly, use `AppHeaderActions` to group right-side items.
- `AppHeaderSearch` expects consumer-driven data: wire `onSearch` to your query library (tanstack query, SWR, etc.) and pass results back as `data`. The component does no fetching.
- Groups render in order. Use `loading: true` on a group to show a pulse bar while results load for that section.
- `AppHeaderSearch` list only shows when the input has text (`data` is defined) — empty input hides the list entirely.
- Group items are memoized on `[data]` — list rendering does not re-run on every keystroke, only when fetched data changes.
- Notification items: use [`AppHeaderNotificationItem`](#appheadernotificationitem) — don't hand-write `.gsl-notif-popover__*` divs.
- There's no `AppHeaderProfile` component — the header profile trigger is [`ProfilePopover`](/profile-popover) itself, with `user`/`variant` for the compact avatar + name/role + chevron trigger. See that page for its full prop table, `AppUser`, and the built-in theme toggle.
- All header components accept `className` and pass it through to their root element for Tailwind/consumer overrides.
- The trigger and popover header avatars default to the normal per-name gradient. Set `--gsl-profile-avatar-bg` (unset by default) on any ancestor to override the background for both — see [`ProfilePopover`](/profile-popover).
- Profile popover opens at `side="top"`, `align="end"`, `sideOffset={8}` — see [`ProfilePopover`](/profile-popover) for the shared implementation.
- `variant="plain"` sets `--gsl-app-header-bg`/`--gsl-app-header-color` on the root — override either on `.gsl-app-header--plain` (or an ancestor) to re-theme without extra CSS. Notification button, profile hover, and `AppSwitcher` trigger all inherit the on-primary foreground automatically.
- **Mobile collapse (automatic):** when `AppHeader` is rendered inside an `AppLayout` (i.e. a `SidebarProvider` is present) and the viewport drops below the sidebar's mobile breakpoint (768px by default), it collapses to a hamburger button on the left — which toggles the sidebar drawer via `SidebarTrigger`'s same `toggle()` — plus, on the right, only the `AppSwitcher` (if one is present) and the profile trigger (`ProfilePopover`, identified by its `componentId`). Branding, search, notifications, and any other actions are dropped. Nothing to opt into — it's derived entirely from the existing children and sidebar context. `AppHeader` used standalone (no `SidebarProvider`) is unaffected and always renders its full children.
