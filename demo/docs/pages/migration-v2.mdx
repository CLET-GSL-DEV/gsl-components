# Migration guide: v1 â†’ v2

This walks through every breaking change and deprecation between `1.22.0` and the current version,
and how to adopt the new design system. See the [Changelog](/docs/changelog) for the full, dated
list â€” this page groups the same changes by what you need to *do* about them.

## The new design system, live

The demo app's root route (`/`, `Dashboard2Page`) is the reference implementation of the new
design system â€” it's what a from-scratch app should look like today. `/legacy` is a frozen
snapshot of the old v1 look (old red branding, old component variants), kept only so the two can
be compared side by side via the version dropdown in its sidebar.

If you're deciding whether a screen "looks right" after upgrading, compare it against `/`, not
against your memory of the old library.

## 1. `AppHeaderProfile` is deprecated â€” migrate to `ProfilePopover` when convenient

**This is a deprecation, not a breaking change.** `AppHeaderProfile` still works exactly as
before â€” it's now a thin wrapper around `ProfilePopover` â€” so existing code keeps compiling and
rendering unchanged. There's no forced migration and no removal date. That said, `ProfilePopover`
is where all new features land, so prefer it directly for new work and migrate existing usages
opportunistically.

`AppHeaderProfile` was the same popover menu as `ProfilePopover` with a second name and a second
prop surface â€” now there's one real implementation. `ProfilePopover` gained `user` and `variant`
props that build the exact same compact avatar + name/role + chevron trigger `AppHeaderProfile`
renders.

```diff
-import { AppHeaderProfile } from "@rfdtech/components";
+import { ProfilePopover } from "@rfdtech/components";

-<AppHeaderProfile user={user} variant="avatar">
-  <RoleSelect roles={roles} selectedRole={role} onClickRole={setRole} />
-</AppHeaderProfile>
+<ProfilePopover user={user} variant="avatar">
+  <RoleSelect roles={roles} selectedRole={role} onClickRole={setRole} />
+</ProfilePopover>
```

`items`, `loading`/`loadingLabel`, `onSignOut`, and the `children` slot carry over unchanged.

Two things don't carry over as-is when you do migrate:

- **`onProfileClick`/`onSettingsClick`/`onHelpClick` have no `ProfilePopover` equivalent.**
  `AppHeaderProfile` hardcodes a "My Profile" / "Account Settings" / "Help & Support" menu built
  from those three callbacks. `ProfilePopover` renders nothing in that section unless you pass
  `items` â€” build the same three rows yourself:

  ```tsx
  <ProfilePopover
    user={user}
    variant="avatar"
    items={[
      { icon: <User size={20} />, label: "My Profile", onClick: goToProfile },
      { icon: <Settings size={20} />, label: "Account Settings", onClick: goToSettings },
      { icon: <HelpCircle size={20} />, label: "Help & Support", onClick: goToHelp },
    ]}
    onSignOut={signOut}
  />
  ```

- **`headerAction` no longer does anything**, on both components. The popover header now always
  shows a built-in light/dark theme toggle when a `ThemeProvider` is in the tree (and nothing when
  it isn't) â€” it's no longer a prop you pass, override, or opt out of. `AppHeaderProfile` still
  *accepts* `headerAction` for source compatibility, but silently ignores it â€” delete it whenever
  you touch the surrounding code, migration or not.

If you're still on `variant="basic"` from before `2.0.0`, that's a separate, older breaking change
(not part of this deprecation) â€” only `"full"` and `"avatar"` remain on both components.

## 2. `AppItem.icon` is now `SystemAppIcon` or `AppIconTile` only

`AppSwitcher`'s `AppItem.icon` used to accept `ReactNode | string` â€” a raw image URL, an emoji, or
any arbitrary node. It's now a closed set: `<SystemAppIcon name="..." />` (auto-generated
initials tile) or `<AppIconTile name="...">custom glyph</AppIconTile>`. This is intentional â€”
every app icon now renders on the shared 9-gradient / 9-overlay tile system, with no per-app
opt-out.

```diff
 const apps = [
-  { id: "analytics", name: "Analytics", icon: "https://cdn.example.com/analytics.png" },
+  { id: "analytics", name: "Analytics", icon: <SystemAppIcon name="Analytics" /> },
-  { id: "custom", name: "Custom App", icon: <MyCustomLogo /> },
+  { id: "custom", name: "Custom App", icon: <AppIconTile name="Custom App"><MyCustomLogo /></AppIconTile> },
 ];
```

## 3. `AppSwitcher` now caps the grid at 6 apps by default

The new `maxItems` prop defaults to `6`. If your app grid previously rendered more than 6 apps,
it's now silently truncated. Pass `maxItems` explicitly to keep the old behavior:

```diff
-<AppSwitcher apps={apps} />
+<AppSwitcher apps={apps} maxItems={apps.length} />
```

Before reaching for `maxItems`, though, consider whether this is a good moment to switch to
[`Launchpad`](#6-prefer-launchpad-over-the-deprecated-appswitcher) instead â€” see Â§6 below.

## 4. `Table`: selection-only tables lose the kebab actions column

Previously, a `Table` with `selectable` but no `rowActions` still rendered a kebab actions column
containing just a Select/Deselect toggle. Selection is already handled by the checkbox column, so
that empty kebab column no longer renders. If you relied on the Select/Deselect entry living in
the kebab menu, add it as an explicit row action instead:

```diff
 <TableContent
   selectable
   selectedIds={selected}
   onSelectionChange={setSelected}
+  rowActions={[
+    { id: "select", label: isSelected ? "Deselect" : "Select", onClick: toggleRow },
+  ]}
   ...
 />
```

## 5. Rebrand: new default colors and typography

If your app doesn't set its own `--gsl-*` token overrides, two visual defaults changed:

- **Brand colors**: `--gsl-primary`/`--gsl-primary-light` moved from red (`#dc2626`/`#ef4444`) to
  navy (`#051b2c`/`#4a7fa8`). A new `--gsl-secondary` (gold, `#c8a24b`) token was added and is now
  the default `SidebarLink` active-link color.
- **Typography**: new `--gsl-font-header` (Roboto Serif) and `--gsl-font-body` (Lato) tokens, shipped
  via two new bundled dependencies (`@fontsource/lato`, `@fontsource/roboto-serif`), now back
  nearly every component's heading and body text â€” each component defines its own
  `--gsl-<component>-font`/`-title-font`/`-value-font` variable defaulting to one of the two.
  Previously everything inherited the single system-font stack (`--gsl-font`). This also grows
  your CSS bundle by the bundled font weight files.

**If you want to keep the old red/system-font look**, override the tokens yourself â€” either in
plain CSS or via [`gslTheme()`](/docs/theme):

```tsx
import { gslTheme } from "@rfdtech/components";

gslTheme({
  all: {
    primary: "#dc2626",
    primaryLight: "#fef2f2",
  },
});
```

```css
:root {
  --gsl-font-header: var(--gsl-font);
  --gsl-font-body: var(--gsl-font);
}
```

**If you want the new look, you don't need to do anything** â€” it's already the default. See
`.legacy-theme` in `demo/demo.css` for exactly which tokens the demo app overrides to reproduce
the old red look on its `/legacy` route, if you want a reference for a similar toggle.

## 6. Prefer `Launchpad` over the deprecated `AppSwitcher`

`AppSwitcher` is deprecated and no longer maintained â€” no removal planned, but no new features or
design updates either. [`Launchpad`](/docs/launchpad) is a separate, from-scratch component that
covers the same "switch between apps" job with a much smaller, more opinionated API, and it's what
new work should use.

`Launchpad` is **not** a drop-in prop-for-prop replacement â€” it's a fully independent component
tree (separate types, CSS classes, hook), so switching means re-authoring the app list and trigger,
not renaming an import:

```diff
-import { AppSwitcher, SystemAppIcon } from "@rfdtech/components";
+import { Launchpad, SystemLaunchpadIcon } from "@rfdtech/components";

 const apps = [
-  { id: "analytics", name: "Analytics", icon: <SystemAppIcon name="Analytics" /> },
+  { id: "analytics", name: "Analytics", icon: <SystemLaunchpadIcon name="Analytics" /> },
   // ...
 ];

-<AppSwitcher apps={apps} maxItems={apps.length} footer={<CustomFooter />} />
+<Launchpad apps={apps} onAppSelect={(app) => navigate(app.href)}>
+  <RoleSelect roles={roles} selectedRole={role} onClickRole={setRole} />
+</Launchpad>
```

What's different, concretely:

- **Icon type is closed to `SystemLaunchpadIcon`/`LaunchpadIconTile`** â€” `AppSwitcher`'s
  `SystemAppIcon`/`AppIconTile` elements don't type-check as a `LaunchpadApp.icon`, even though
  they look identical. Swap the import, not just the usage.
- **Grid size isn't configurable.** `AppSwitcher`'s `maxItems`/`columns` are gone â€” the popover
  grid is a fixed 3Ă—3 (9 apps), full stop. The built-in "See all" button (always rendered, no prop
  to hide it) opens an uncapped 5-column modal showing every app in `apps` â€” that's the only way
  to see more than 9.
- **No generic `footer` slot.** `Launchpad`'s only composable slot is `children`, and it only
  accepts a single `RoleSelect` element, rendered below the built-in "See all" button. A custom
  footer (arbitrary content, multiple elements, non-`RoleSelect` content) has no equivalent â€” keep
  using `AppSwitcher` if you genuinely need one.
- **Title, tooltip, and close-on-select behavior are fixed.** No `title`/`triggerLabel`/
  `loadingLabel`/`onSeeAll`/`seeAllLabel`/`placement`/`closeOnSelect` props â€” the trigger always
  reads "Open Launchpad", the popover/modal title always reads "Launchpad", and selecting an app
  always closes the popover.

**Stay on `AppSwitcher` for now if you need:** a flexible column count, an uncapped or
differently-capped grid without the "See all" modal, or a fully custom `footer` â€” `Launchpad`
doesn't cover those (yet). Check with the team before building new custom-footer `AppSwitcher`
usage, since that's the main gap keeping `Launchpad` from being a universal replacement.

## Adopting the new design system

**This section is the real point of "v2."** Everything in Â§1â€“Â§6 above is about not breaking when
you upgrade the dependency. This section is about actually *looking like* the new design system â€”
matching `/` (`Dashboard2Page` + its `DemoLayout2` shell), not just staying green on a version bump.
None of it is required to keep existing screens compiling and rendering. All of it is what new
screens, and any screen you're meaningfully touching, should be built with. If a screen still has
the old bordered-panel-everywhere look after you've "migrated" it, it isn't actually migrated yet â€”
compare it against `/` directly, section by section, not from memory.

### Layout shell: `AppLayout variant="stacked"`, `AppHeader variant="plain"`, `Sidebar variant="plain"`

The reference shell (`DemoLayout2`) composes three layout-level variant choices together â€” they're
designed to be adopted as a set, not individually:

```diff
-<AppLayout>
+<AppLayout variant="stacked">
   <AppHeader variant="plain">
     ...
   </AppHeader>
   <AppSidebar>
     <SidebarOverlay />
-    <Sidebar>
+    <Sidebar variant="plain">
       ...
     </Sidebar>
   </AppSidebar>
   <AppBody>
     <Outlet />
   </AppBody>
 </AppLayout>
```

- **`AppLayout variant="stacked"`** â€” header spans the full width across the top; sidebar and
  content sit side by side below it. The default (`"default"`) instead runs the sidebar full-height
  down the left with the header only spanning the content column. `/` uses `"stacked"`.
- **`AppHeader variant="plain"`** â€” square corners, solid `--gsl-primary` background, on-primary
  text, reads as one continuous top bar with the branding block. Already covered above, repeated
  here because it's part of the same shell decision, not standalone.
- **`Sidebar variant="plain"`** â€” transparent background plus a right border, instead of the
  default's own panel surface/background. Paired with the `"plain"` header and `"stacked"` layout,
  this is what removes the boxed-in, panel-inside-panel look of the old default shell.

**`SidebarGroup collapsible`** (already listed below as a standalone preference) is also part of
this shell in practice â€” `/`'s nav groups are all `collapsible` accordion sections, not flat lists.

**`SidebarFooter` should host a `ProfilePopover`** â€” don't leave the sidebar without a footer, and
don't hand-roll a footer profile row. `/` renders `SidebarFooter` with `ProfilePopover` in
`fullName`/`email` mode (not `user`/`variant` â€” that mode is for the compact header trigger, see
"One role switcher" below), so the full name, email, and role always have a stable,
real-estate-rich home even when the header trigger stays down to an icon.

**Branding lives in the header (`AppHeaderBranding`), not the sidebar.** `/`'s `Sidebar` has no
`SidebarHeader`/`SidebarBrand` at all â€” the logo and product name live exclusively in `AppHeader`
via `AppHeaderBranding` (`logo`/`title`/`subtitle`), positioned first inside the header, before
`AppHeaderActions`:

```tsx
<AppHeader variant="plain">
  <AppHeaderBranding
    logo={<img src="/logo.png" alt="" width={28} height={28} />}
    title="YOUR PORTAL"
    subtitle="Your Product Name"
  />
  <AppHeaderActions>...</AppHeaderActions>
</AppHeader>
```

`SidebarHeader`/`SidebarBrand` are **not deprecated** â€” they're still valid, documented components
for apps that want sidebar-only branding with no `AppHeader` branding slot in use. The point isn't
that they're wrong, it's that the same logo and product name shouldn't appear twice on screen at
once. If your layout is adopting `AppHeaderBranding` in the header, remove the sidebar's own
`SidebarHeader`/`SidebarBrand` block â€” don't keep both.

### Use built-in `loading` states â€” don't hand-roll placeholders

Nearly every component in the shell and on `/` itself supports a real `loading` prop that renders a
proper shimmering skeleton â€” `/` uses every one of these, driven from real query loading flags
(`useMockQuery` in the demo; your real data-fetching hook in an app):

| Component | Loading prop(s) | Replaces |
|-----------|------------------|----------|
| `Launchpad` / `AppSwitcher` | `loading` | A disabled/empty trigger, or rendering nothing until apps arrive |
| `AppHeaderNotifications` | `loading` | An empty popover, or a hand-rolled spinner |
| `ProfilePopover` | `loading` + `loadingLabel` | Rendering `"..."` or nothing for the name/email/avatar |
| `SidebarLink` | `loading` + `loadingLabel` | Not rendering nav items at all until data resolves, or a generic full-page spinner |
| `MetricCard` | `loading` + `loadingLabel` | A hand-rolled `"..."` or `"N/A"` string in `value`/`description` |
| `TableContent` | `loading` + `loadingRows` | An empty table, a spinner overlay, or hiding the table until data arrives |

The `SidebarLink` pattern specifically â€” render a handful of `loading` links in place of the real
nav while it's being fetched, not a full-sidebar spinner or an empty sidebar:

```tsx
{navLoading ? (
  <SidebarGroup>
    {Array.from({ length: 6 }).map((_, i) => (
      <SidebarLink key={i} loading icon={<span />}>Loading</SidebarLink>
    ))}
  </SidebarGroup>
) : (
  navGroups.map((group) => /* real nav */)
)}
```

If you find a component rendering `value={data ?? "..."}`, `description={loading ? "Checking..." :
...}`, a hand-rolled `<Spinner />`, or simply not rendering until data resolves â€” and that
component is in the table above â€” replace it with the real `loading` prop instead. This is
independent of the layout-shell and table-variant changes above; audit page-level usages too, not
just the shell files.

### One role switcher, driven consistently across all three places it can appear

`RoleSelect` can be composed as `children` in exactly three components:
`ProfilePopover` (header), `ProfilePopover` (sidebar footer), and `Launchpad`. If your app has
role-switching at all, **wire all three from the same `roles`/`selectedRole`/`onClickRole`
state** â€” not just whichever one you happened to add first. A user shouldn't find the role
switcher in the sidebar footer but not in the header, or in either profile popover but not in
Launchpad. Each location needs its own `<RoleSelect>` element (the prop only accepts one element
per slot, so you can't literally share a single node across three places), but all three should
read from and write to the same state so switching role anywhere is reflected everywhere:

```tsx
const [selectedRole, setSelectedRole] = useState(roles[0].id);
const roleSelectProps = { title: "View as", roles, selectedRole, onClickRole: (r) => setSelectedRole(r.id) };

// Header
<ProfilePopover variant="avatar" user={user} items={items} onSignOut={onSignOut}>
  <RoleSelect {...roleSelectProps} />
</ProfilePopover>

// Sidebar footer
<SidebarFooter>
  <ProfilePopover fullName={user.name} email={user.email} items={items} onSignOut={onSignOut}>
    <RoleSelect {...roleSelectProps} />
  </ProfilePopover>
</SidebarFooter>

// Launchpad (header)
<Launchpad apps={apps} onAppSelect={onAppSelect}>
  <RoleSelect {...roleSelectProps} />
</Launchpad>
```

**Header `ProfilePopover` prefers `variant="avatar"`** (icon only, no name/role/chevron) â€” `/` uses
this, specifically *because* the full name/role/email are already given a proper home in the
`SidebarFooter` popover (`fullName`/`email` mode). Don't use `variant="full"` in the header if
you're also rendering a `SidebarFooter` popover â€” that duplicates the same name/role text in two
places on screen at once. Reserve `variant="full"` for headers that have no sidebar footer at all.

### Table family

`TableContent`, `TableFilter`, and `TableFooter` are meant to be adopted together, not
independently â€” `/` uses all three of these on every table:

| Component | Prefer | Over |
|-----------|--------|------|
| `TableContent` | `variant="panel"` | `variant="default"` |
| `TableFilter` | `variant="spread"` | `variant="popover"` (default) |
| `TableFooter` | `className="gsl-table__footer--no-border"` | no className (default top border) |

```diff
 <Table paramPrefix="members">
   <TableHeader>
     <TableSearch placeholder="Search members..." />
-    <TableFilter>
+    <TableFilter variant="spread">
       <Dropdown name="role" ... />
     </TableFilter>
   </TableHeader>
-  <TableContent columns={columns} data={data} rowKey={...} />
+  <TableContent variant="panel" columns={columns} data={data} rowKey={...} />
-  <TableFooter>
+  <TableFooter className="gsl-table__footer--no-border">
     <TablePagination ... />
   </TableFooter>
 </Table>
```

`TableFooter`'s `--no-border` class is a real, stable class from the library's own stylesheet
(`.gsl-table__footer--no-border { border-top: none; }`), not a demo-only hack â€” it's the finishing
touch that makes the `"panel"`/`"spread"` pairing read as one continuous surface instead of a
bordered box with another bordered strip glued to the bottom.

### Other preferred variants

| Component | Prefer | Over |
|-----------|--------|------|
| `MetricCard` | `variant="outline"` | `variant="default"` |
| `Tabs` | `variant="pill"` | `variant="default"` |

`SectionHeader` (with `SectionTitle`/`SectionDescription`/`SectionActions`) is also worth adopting
for page headers â€” it's what `/` uses for its title + description + action-buttons row instead of
hand-rolled `<div>`s, though it's a new addition rather than a variant of something older, so
there's no "instead of" to name here.

None of the variants on this page require a code change to keep existing screens working â€” they're
additive, not replacements for the defaults, and there's no deadline to migrate existing screens off
the old ones. But treat this whole section, not just the breaking changes above it, as the actual
definition of "migrated to v2" for any screen you're building fresh or substantially touching.
